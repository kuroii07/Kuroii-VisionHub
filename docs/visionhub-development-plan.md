> 本文档是 VisionHub Studio 后续计划开发的入口文档。后续进入较大优化、版本开发、发布构建或 GitHub 推送前，先读 `AGENTS.md`，再读本文档，最后按具体模块补读专项文档。

## 1. 当前基线

- Current app version: `0.4.5`
- 当前平台：Windows 优先
- 当前发布策略：正式发布准备后移到 `v1.0` 前；`0.3.x` 进入收口补丁，`0.4.x` 进入日常可用性和稳定性增强
- 当前主方向：中转站 / 聚合 API 优先，官方 API 和本地模型保留清晰规划入口
- Current focus: `0.4.5` global experience and performance QA: UI QA baseline, accessibility names, long-text guards, empty/error states, and large-list performance guards without redesigning AI Create Desk.

## 3. 版本、提交与推送策略

### 3.1 需要更新版本号的情况

以下属于“大优化 / 大修改”，必须更新更细版本号，并在验证后提交推送到 GitHub：

- 新增或重构一个完整功能模块。
- 修改 Provider 请求协议、图生图映射、凭据迁移、生成记录结构等核心数据流。
- 改动作品画廊、灵感中心、图生图工作台等主要页面的信息架构。
- 引入新的持久化文件、迁移逻辑、批量操作或恢复机制。
- 发布新的 release exe、绿色测试版、MSI 或 NSIS 安装包。
- README、release notes、版本号、安装包名称需要同步的发布型改动。

版本更新必须同步检查：

- `package.json`
- `package-lock.json`
- `src-tauri/Cargo.toml`
- `src-tauri/tauri.conf.json`
- `src-tauri/Cargo.lock`
- README 当前状态或版本说明
- 应用内设置页 / 系统信息中显示的版本

### 3.2 不需要更新版本号的情况

以下属于“小优化 / 小修改”，默认不更新版本号，也不提交推送：

- 文案微调、错别字、小范围样式间距调整。
- 不改变交互和数据结构的局部 UI 清理。
- 临时调试日志、开发辅助脚本、本地验证笔记。
- 只影响当前测试环境、不准备保留为版本成果的改动。

如果小修改最终累计成稳定体验改进，可以在下一个版本号里一起整理提交。

### 3.3 文档改动规则

- 计划类文档可以提交推送，但不单独更新应用版本号。
- 与版本发布绑定的 README、release notes、安装说明必须跟随对应应用版本一起提交。
- 本文档变更时，需要确认是否影响后续版本顺序；如果影响，要同步更新 `README.md` 的后续路线入口。

### 3.4 GitHub 同步规则

大版本或阶段版本完成后的默认流程：

1. 运行相关验证。
2. 在路线文档和 README 中将完成项用删除线标记，并补充“已完成 / 已落地 / 已收口”等状态。
3. 如果阶段完成后还有小修改、小优化，先继续在本地修到稳定，不急于更新版本号或推送。
4. 用户确认某个细版本最终收口后，再统一更新版本号、README 当前状态、验证记录，并提交推送到 GitHub。
5. 正式发布准备、release notes、安装包、SHA256、迁移说明和对外发布清单整体后移到 `v1.0` 前。

默认不把 `node_modules/`、`dist/`、`src-tauri/target/`、生成图片、AppData、API Key、安装包提交进源码仓库。安装包如果要发布，优先作为 GitHub Release Asset。

### 4.1 路线分层说明

- `0.3.10` is closed as the final 0.3 patch baseline; `0.4.1` closed the Prompt workflow scope; `0.4.2` closed Provider self-check hardening without adding unverified provider templates.
- `0.4.x` 是可用性增强阶段，重点让现有功能形成稳定工作流：作品画廊整理、Prompt 复用、Provider 稳定、本地模型增强、数据迁移和全局 QA。
- `0.5.0` 是发布候选阶段，目标是把自用软件整理到可迁移、可复现、可打包验证的状态。
- `v1.0` 前只处理发布、迁移、安装包、签名风险和 GitHub Release 边界，不再混入大功能。

## 5. 阶段计划

### 5.1 `0.2.2` 平台能力矩阵 V2

目标：

- 让用户一眼知道每个平台“能做什么、现在能不能用、为什么不能用”。
- 区分中转站 / 聚合 API、官方 API、本地模型三条路线。
- 待接入模板保持只读规划状态，避免误导。

主要任务：

- 定义平台能力展示模型：文生图、图生图、多参考图、Responses、Images、OpenAI-compatible、官方协议、本地服务。
- 在平台接入页增加能力矩阵区域或矩阵详情视图。
- 为每个 Provider / 模板设置状态：已接入、可配置、实验中、待接入、本地规划。
- 为不可用能力提供简短原因和后续路线，不提供会误导的保存启用或试生图按钮。
- 保持 `custom-http-provider` 作为聚合站默认主入口。

验收标准：

- 用户不需要读长说明也能区分“可真实生成”和“仅规划展示”。
- OpenAI 官方和 OpenAI-compatible 聚合站文案不混淆。
- 本地模型入口不会影响当前中转站主流程。
- 浅色 / 深色模式下矩阵可读，长平台名不重叠。

### 5.2 `0.2.3` 图生图协议适配 V2

状态：已完成第一轮落地。`0.2.3` 已把图生图映射从后端隐式逻辑提升为可配置项，并在生成结果 raw 中记录 `visionhub_protocol_mapping` 摘要。能力矩阵 V2 同步改为折叠查看，避免遮挡右侧配置详情。

目标：

- 将当前图生图参考图管理能力，逐步映射到真实 Provider 请求。
- 支持不同中转站对参考图字段和 endpoint 的差异。

主要任务：

- 梳理当前 `ReferenceImage`、角色标记、顺序、最多 4 张规则与后端请求结构。
- 扩展 Provider profile 或协议配置，描述参考图字段名、图片数组规则、endpoint path、Responses input_image 结构。
- 将参考强度、保留构图、风格迁移等偏好从记录层接入请求层。
- 对不支持图生图的 Provider 给出明确禁用和说明。
- 生成记录保存实际使用的参考图角色、顺序、协议路径和映射结果摘要。

验收标准：

- 聚合站 / OpenAI-compatible 图生图配置有明确映射入口。
- 多参考图顺序和角色不会在请求前后丢失。
- 协议不匹配时错误可读，不只显示底层 HTTP 报错。
- 失败记录能说明是鉴权、模型、协议路径还是参考图字段问题。

### 5.4 `0.2.5` AI 创作功能区 V3

状态：已完成基础落地并进入 `0.2.5`。本轮已完成提示词润色窗口 V2、模板灵感 V2 和复用记录 V2：润色弹窗支持左右编辑、恢复原文、交换左右、替换、追加和复制；模板灵感扩充到多场景模板并支持用途筛选；复用记录支持关键词、平台、模型、类型、状态筛选、排序、筛选计数、重置筛选和删除软件记录。模板收藏、最近使用模板、用户自定义模板以及导入 / 导出自定义模板转入后续增强计划。

目标：

- 把 AI 创作页底部三个高频弹窗从“能查看 / 能润色”升级为“可编辑、可复用、可管理”的创作辅助区。
- 优先解决提示词润色、复用记录和模板灵感的实际操作效率，不改 Provider 凭据、生成记录结构或主生成协议。

主要任务：

- 提示词润色窗口 V2：原提示词展示区和润色结果展示区从只读改为可编辑。
- 润色结果可编辑后，支持替换当前 Prompt、追加到当前 Prompt、复制结果和保留用户手动修改。
- 对齐“原提示词 / 字符数量”和“润色结果 / 字符数量”，保证左右面板标题、字符数和操作按钮横向稳定。
- 复用记录窗口 V2：历史记录增加复用、追加、复制和删除记录能力。
- 复用记录增加过滤与排序：关键词、平台、模型、文生图 / 图生图、有参考图、最近生成时间。
- 删除历史记录只删除软件记录，不默认删除磁盘图片；批量删除或影响文件的动作必须二次确认。
- 模板灵感窗口 V2：扩充模板库，覆盖商业海报、角色设定、电商主图、产品摄影、游戏资产、社媒封面、图生图改写、中文平台等方向。
- 模板灵感增加分类筛选、收藏常用模板、一键替换、一键追加和模板适用场景说明。
- 三类弹窗统一空状态、加载状态、按钮层级、字符统计、防溢出和暗色 / 浅色模式样式。

补充建议：

- 复用记录可以增加“标记常用 Prompt”和“只看成功生成记录”，避免失败 Prompt 混入常用列表。
- 模板灵感可以增加“最近使用模板”和“按当前模式推荐模板”，文生图 / 图生图展示不同默认模板。
- 提示词润色窗口可以增加“恢复原文”和“交换左右内容”能力，方便多轮编辑。

验收标准：

- 用户可以在润色弹窗内直接编辑原提示词和润色结果，不需要回到主输入框反复复制。
- 字符数量、标题和操作按钮在中英文、长 Prompt 和窄窗口下不错位。
- 复用记录至少支持复用、追加、复制和删除，删除不会误删磁盘图片。
- 模板数量明显多于旧版 4 个，并能按用途快速筛选。
- 三类弹窗在浅色 / 暗色模式下都没有明显遮挡、重叠或按钮不可达。

### 5.6 `0.2.7` 图生图工作台体验 V2

状态：0.2.7 已完成。参考图管理、多参考修复、错误诊断接入、真实接口文生图/1-4 张参考图生图全测通过。。当前先做用户可感知的缩略图顺序、角色状态、导入反馈和弹层层级，不改 Provider 凭据、主生成协议或历史数据。

  - 4 张参考图、长 Prompt、长模型名、失败导入同时出现时是否稳定。

- 通用
  - 不能只检查暗色模式。
  - 不能只看有数据状态，也要看空状态和失败状态。
  - 不用大段说明文字堆主界面，优先用标题、状态、tooltip 和详情入口承载解释。

## 7. 每个版本的执行清单

开始前：

- 读取 `AGENTS.md` 和本文档。
- 读取对应专项文档。
- `git status --short --branch`，确认当前工作区状态。
- 明确本次版本号、主目标和不做事项。

实现中：

- 优先复用现有类型、service、store、Tauri command 和 CSS token。
- 不为局部问题追加散乱样式；先定位重复选择器和冲突层级。
- 不改系统代理、AiMaMi、Clash/VPN、DNS、全局执行策略。
- 不清空用户图库、历史记录、配置、缓存或凭据。

验证：

- 文档改动：至少运行 `git diff --check`。
- 前端 / UI 改动：运行 `npm.cmd run build` 和 `python .\scripts\smoke_check.py`。
- Tauri / Rust 改动：补跑 `cargo check`。
- 发布型改动：运行项目验证脚本和 `npm.cmd run tauri:build`。
- UI 改动：实际启动软件检查关键页面，尤其浅色模式。

收尾：

- 完成一个路线项后，在本文档和 README 中用删除线划掉已完成事项，并补充状态。
- 如果还有小修改、小优化，继续本地迭代，不急于更新版本号或推送。
- 用户确认该细版本最终收口后，再统一更新版本号、README、验证记录，并提交推送 GitHub。
- `v1.0` 前再集中处理 release 构建、安装包、迁移说明、release notes 和 SHA256。
- 最终回复写清改动文件、验证命令、是否推送、未完成事项。
